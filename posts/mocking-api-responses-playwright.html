Testing UI Number Formatting by Mocking Backend Responses with Playwright and Python

A practical approach to validating how UI displays numeric values by intercepting backend responses and returning controlled test data.

Why mock backend responses?

In many real-world scenarios, the backend returns valid data, but the real issue is how the frontend formats and displays that data to the user.

For example, the backend may return 1000000, while the CRM is expected to display 1M in the interface. In this case, the goal is not to validate backend calculations, but to verify UI formatting logic.

Test idea

Instead of depending on real backend data, the Playwright test intercepts the API request and returns a mocked JSON response. This makes it possible to validate edge cases, control input values, and check exactly what the UI renders.

Mocked response example

MOCK = {
    "totalTopUps": 1010,
    "totalRedemptions": 1499,
    "totalPointsOutWithoutCommission": 1500,
    "totalFLCommission": 535342.22,
    "totalOutWithCommission": 1999,
}

Intercepting the request

def handler(route):
    route.fulfill(
        status=200,
        content_type="application/json",
        body=json.dumps(MOCK),
    )

page.route("**/api/loyalty/rewards/charges/summary*", handler)

UI assertions

After intercepting the backend response, the test verifies that the UI renders the expected formatted values in the corresponding fields. 


                    assert page.locator("[data-testid='total-top-ups']").inner_text() == "1K"
                    assert page.locator("[data-testid='total-redemptions']").inner_text() == "1.4K"
                    assert page.locator("[data-testid='total-out-with-commission']").inner_text() == "2K"

Each assertion checks two things:

  • the UI element exists
  • the displayed value matches the expected formatted representation

By mocking the backend response, the test can reliably validate how the frontend formats and displays numeric values.

Why not use real backend data?

In theory, the same UI formatting could be tested using real backend data. However, relying on real environments often makes UI tests unstable and difficult to maintain.

Test environments rarely contain the exact values needed to verify formatting boundaries such as 999 → 1K or 999999 → 1M. Preparing such data manually may require database manipulation or additional backend setup.

Mocking the response solves this problem. The test controls the exact values returned by the API and can immediately validate how the frontend renders them.

This makes the test:

  • deterministic
  • independent from backend state
  • easier to reproduce and debug

Test design techniques

To generate useful test cases for numeric formatting, I would use Boundary Value Analysis and Equivalence Class Partitioning. These techniques help cover display rules around transitions such as 999 → 1K and 999999 → 1M.

For example, when testing compact number formatting, important boundaries include values around 999, 1000, 999999, and 1000000.
